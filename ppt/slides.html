Context

The Real-World Challenge

Organizations process thousands of receipts and invoices daily.

  • Manual data entry is slow, error-prone, and expensive
  • Receipts are unstructured — different layouts, fonts, languages
  • Businesses need: structured fields + ability to query them naturally

What we need to build

Extract → Total, Date, Vendor, Receipt ID

Store → Structured JSON in database

Query → "What is the total?" from extracted data only

Why it's hard

OCR noise on real receipts

Blur, low contrast, crinkled paper

No fixed layout

Every receipt looks different

LLMs hallucinate if not grounded

Will invent plausible-looking numbers

Data

Dataset & Preprocessing

CORD Dataset

naver-clova-ix/cord-v1 — Hugging Face

  • Receipt images with OCR text and labeled fields
  • Word-level bounding boxes (x0, y0, x1, y1)
  • Ground truth: menu items, prices, totals, vendor, date
  • Token-level annotations aligned to words

Used subset: 200 training samples (efficient for free GPU)

Preprocessing Steps

1. Load subset via datasets library
2. Parse ground_truth JSON → extract field values
3. Map to BIO tags: B-TOTAL, I-TOTAL, B-DATE, B-VENDOR…
4. Normalize bboxes to 0–1000 range (LayoutLMv3 format)
5. Tokenize with LayoutLMv3Processor (image+words+boxes)
6. Align labels to first subtoken (others → -100)
7. Train/validation split: 80/20
Attempt 1

Text-Only NER ✗

DistilBERT / BERT Text-Only NER

Fine-tune standard text NER on OCR text only

  • Extracted raw OCR text from CORD
  • Mapped words to BIO labels
  • Fine-tuned distilbert-base-uncased
  • Evaluated with seqeval F1

Result

F1 score was poor — model struggled to identify fields reliably

Why it failed

"334,000" looks identical whether it's a price, total, or phone number

Confused menu item prices with totals constantly

Vendor name detection was near-random

Text-only = no spatial awareness

"Total" at the bottom of a receipt means something different from "Total" in a column header. Pure text NER cannot distinguish these.

→ Layout-aware model required.

Attempt 2

LayoutLMv3 ✓

Why LayoutLMv3

  • Jointly encodes text tokens + bounding boxes + image patches
  • Pre-trained on document understanding tasks
  • Understands spatial relationships — "bottom right = likely total"
  • State-of-the-art on CORD, FUNSD benchmarks

Export: ONNX via optimum-cli

3–5× faster CPU inference vs PyTorch

Pushed to HuggingFace: ShaibinkB/cord-layoutlmv3-onnx

Key Observations

MENU & PRICE: most common labels → learned well

TOTAL works when layout is clean

DATE & RECEIPT_ID: rare in 200-sample subset → underfit

Expected & acceptable — LLM fallback covers weak fields

Real receipt behavior

• Menu items & prices: confidence >0.90

• Total: derived from PRICE near "Total" keyword

• Vendor: derived from first MENU span

• Missing fields → Groq fills from OCR text

Strategy

Hybrid Extraction

No single approach is perfect

Stage 1 — LayoutLMv3 NER (always runs first)

Fast: ~1s CPU (ONNX) · Offline · Returns entity spans + confidence

↓

All fields found AND confidence ≥ 0.75?

→ Return NER result directly (method = "ner")

↓

Stage 2 — Groq LLM (llama-3.3-70b-versatile)

Sends full OCR text with structured JSON prompt · ~400–700ms

↓

Groq fails?

→ Stage 3 — Gemini 2.0 Flash (secondary fallback)

Smart Merging Rules

NER confidence ≥ 0.75 → trusted, LLM cannot override

NER confidence < 0.75 → LLM overrides if better

Field completely missing → LLM fills it

Result: Speed + Accuracy + Robustness

Challenges

How They Were Solved

1

Text-only NER failed

DistilBERT couldn't distinguish "334,000" as total vs price

→ Switched to LayoutLMv3 (text + bbox + image)

2

ONNX rejected bbox inputs

ORTModelForTokenClassification rejected bbox and pixel_values

→ Used ORTModelForCustomTasks + manual input filtering

3

No TOTAL/VENDOR predictions

200-sample subset underrepresents these labels

→ Smart heuristics: derive from PRICE/MENU + LLM correction

4

Groq model decommissioned

llama3-8b-8192 was deprecated mid-development

→ Migrated to llama-3.3-70b-versatile (better anyway)

5

Google AI SDK deprecated

google.generativeai package end-of-life

→ Migrated to new google-genai SDK + gemini-2.0-flash

6

CORS blocking frontend

Backend on port 3000, Vite on 5173

→ CORS_ORIGIN configurable via .env
